개발 중인 앱의 기능이 뒤집어지는 건 사실 일상이다. 어제의 혁신적인 아이디어가 오늘의 쓰레기통으로 가기도 하고, 기껏 구현해 놓은 기능을 기획이 바뀌었다는 이유로 통째로 들어내기도 한다. 구현 방법 자체가 달라지는 경우도 부지기수다. 이런 역동적인 환경에서 개발자를 가장 괴롭히는 건 코드 수정 그 자체가 아니다. 바로 **이미 쌓여 있는 데이터** 와 **오늘의 새로운 요구사항** 사이의 괴리다.

어제까지는 `이름` 필드 하나면 충분했는데, 오늘 갑자기 성과 이름을 분리해달라고 한다면? 혹은 `나이` 를 숫자로 저장했는데 이제 생년월일을 받아야 한다면? 이미 수만 명의 사용자가 `이름` 에 '홍길동'이라고 저장해 둔 상태에서 무작정 코드를 바꾸면 앱은 터져버린다. 어제의 데이터 포맷과 오늘의 구현 로직이 충돌하기 때문이다. 이 간극을 메우고 데이터를 안전하게 새로운 세상으로 옮겨주는 작업, 그것이 바로 `마이그레이션(Migration)` 이다.

단순히 "코드를 고친다"는 행위를 넘어, 이미 살아 움직이는 생물과 같은 데이터를 다루는 법. 오늘은 `마이그레이션` 의 본질과 왜 우리가 이 번거로운 작업을 숙명처럼 받아들여야 하는지 깊이 있게 파헤쳐 보려 한다.

### 1. 마이그레이션이란 무엇인가?

`마이그레이션` 은 직역하면 '이주' 혹은 '이동'이다. 소프트웨어 엔지니어링 관점에서는 한 운영 환경에서 다른 운영 환경으로 데이터나 소프트웨어를 옮기는 과정을 말한다. 하지만 실무에서 우리가 가장 자주 접하는 건 `데이터베이스 마이그레이션(DB Migration)` 이다.

쉽게 말해 **데이터베이스의 스키마(구조)를 변경하고, 그에 맞춰 기존 데이터를 변환하는 일련의 과정** 이다. 집으로 비유하자면, 사람이 살고 있는 집의 벽을 허물고 방 구조를 바꾸면서도 그 안에 있는 가구들을 망가뜨리지 않고 배치만 새로 하는 작업과 같다. 거주자(사용자)가 불편함을 느끼지 않게 하면서 집(인프라/데이터)을 고치는 고난도 공사인 셈이다.

### 2. 왜 마이그레이션이 필요한가?

단순히 "데이터를 옮긴다"는 개념을 넘어, `마이그레이션` 이 필수적인 이유는 명확하다.

첫째, **데이터 정합성(Data Integrity)** 유지다. 코드만 바꾸고 `DB` 구조를 그대로 두면, 새로운 코드는 존재하지 않는 컬럼을 찾으려다 에러를 낸다. 반대로 `DB` 만 바꾸고 코드를 안 바꾸면 런타임 오류가 발생한다. `마이그레이션` 은 이 둘의 싱크를 맞추는 유일한 방법이다. 데이터는 한 번 꼬이면 복구하는 데 수십 배의 노력이 들기 때문에, 처음부터 엄격하게 관리해야 한다.

둘째, **서비스의 연속성** 이다. 새로운 기능을 배포한다고 해서 기존 사용자들의 데이터를 다 날려버릴 수는 없다. 사용자가 잠든 사이에 혹은 서비스가 돌아가는 도중에 데이터를 안전하게 새 포맷으로 '변신'시켜야 한다. 현대의 웹 서비스에서 '점검 중' 화면을 띄우지 않고 데이터를 옮기는 기술은 시니어 개발자의 핵심 역량 중 하나다.

셋째, **협업과 형상 관리** 다. 현대의 `마이그레이션` 도구들은 소스 코드처럼 버전 관리가 가능하다. "야, 너 DB에 `email` 필드 추가했어?"라고 물어볼 필요 없이, 마이그레이션 파일을 실행하는 것만으로 모든 팀원이 동일한 `DB` 구조를 가질 수 있다. 이는 `CI/CD` 파이프라인 자동화의 초석이 된다.

### 3. 마이그레이션의 장점과 단점

모든 기술이 그렇듯 `마이그레이션` 도 비용이 발생한다. 얻는 것이 있다면 잃는 것도, 혹은 감수해야 할 위험도 있는 법이다.

**장점:**
- **버전 관리와 히스토리**: `DB` 의 변화를 시간 순서대로 추적할 수 있다. 누가, 언제, 왜 이 필드를 추가했는지 기록이 남는다. 문제가 생기면 이전 상태로 되돌리는 `롤백(Rollback)` 이 가능하다.
- **재현성**: 어떤 환경(로컬, 스테이징, 운영)에서도 동일한 `DB` 상태를 만들 수 있다. "내 컴퓨터에서는 잘 되는데?"라는 소리가 나올 확률을 획기적으로 줄여준다.
- **자동화된 배포**: 수동으로 `SQL` 문을 날리는 것보다 실수가 적고, 배포 스크립트에 포함하여 원클릭으로 `DB` 구조 변경부터 코드 반영까지 끝낼 수 있다.

**단점:**
- **복잡도 증가**: 단순한 프로젝트라면 `마이그레이션` 도구를 세팅하는 것 자체가 오버헤드일 수 있다. 스키마가 복잡해질수록 마이그레이션 파일 관리도 일이 된다.
- **데이터 유실 위험**: `마이그레이션` 스크립트를 잘못 작성하면 기존 데이터가 영구적으로 손상될 수 있다. 특히 `DROP COLUMN` 같은 명령어를 포함할 때는 손이 떨릴 수밖에 없다. (항상 백업이 필수인 이유다.)
- **성능 부하와 다운타임**: 수백만 건, 수천만 건의 데이터를 한꺼번에 변경할 때는 `DB` 에 상당한 부하를 주며, 최악의 경우 테이블 락(`Table Lock`)이 걸려 서비스가 마비될 수 있다.

### 4. 실전 사례: 변화하는 요구사항에 대응하기

#### Case 1: 이름 필드 분리하기
기존 `User` 테이블에 `full_name` 이라는 필드 하나만 있었다고 가정하자. 그런데 글로벌 진출을 위해 성(`last_name`)과 이름(`first_name`)을 분리해야 하는 상황이 왔다. 단순히 컬럼을 삭제하고 새로 만들면 기존 데이터는 증발한다.

제대로 된 전략은 **확장 - 이관 - 삭제** 의 단계를 거친다.
1. `first_name` 과 `last_name` 컬럼을 새로 추가한다.
2. 기존 `full_name` 데이터를 읽어와서 로직에 따라 쪼갠 뒤 새 컬럼들에 채워 넣는다.
3. 코드가 새 컬럼을 사용하도록 업데이트하여 배포한다.
4. 일정 기간 데이터가 안정적으로 쌓이는 것을 확인한 후, 더 이상 사용하지 않는 `full_name` 컬럼을 제거한다.

#### Case 2: 데이터 타입의 변경
정수(`INT`)로 관리하던 `ID` 값이 서비스 성장으로 인해 범위를 초과할 것 같아 `BIGINT` 로 바꿔야 한다면? 혹은 `String` 으로 저장하던 날짜 데이터를 `DateTime` 포맷으로 엄격하게 관리해야 한다면? 이때는 단순한 구조 변경을 넘어 데이터의 '형 변환'이 수반된다. 데이터 양이 많을 경우 백그라운드 프로세스로 조금씩 옮기는 작업이 필요할 수도 있다.

### 5. 아키텍처의 거대한 이동: 클라우드와 마이크로서비스

단순한 테이블 구조 변경을 넘어, 최근에는 시스템 전체의 판을 짜는 `아키텍처 마이그레이션` 도 빈번하다.

가장 대표적인 것이 **모놀리식(Monolithic)에서 마이크로서비스(Microservices)로의 전환** 이다. 하나의 거대한 `DB` 에 모든 데이터가 들어있던 시절에서, 서비스별로 `DB` 를 쪼개는 과정은 `마이그레이션` 의 끝판왕이라 불릴 만하다. 이때는 단순 `SQL` 이 아니라, 데이터를 실시간으로 동기화하고 정합성을 맞추기 위해 `CDC(Change Data Capture)` 나 `이벤트 소싱(Event Sourcing)` 같은 고난도 기술이 동원된다.

또한 온프레미스(자체 서버)에서 `AWS` 나 `GCP` 같은 클라우드로 옮기는 **클라우드 마이그레이션** 도 있다. 이는 단순히 서버를 옮기는 것을 넘어, 사용하는 `DB` 엔진의 특성이 달라지거나 네트워크 지연 시간(`Latency`)이 변하는 등의 변수까지 고려해야 하는 거대한 작업이다.

### 6. 무중단 마이그레이션(Zero Downtime)의 마법

사용자가 서비스를 이용하는 도중에 `DB` 구조를 바꾸는 것은 마치 달리는 자동차의 바퀴를 갈아 끼우는 것과 같다. 이를 위해 시니어 개발자들은 `Expand and Contract` 패턴을 사용한다.

1. **Expand (확장)**: 새로운 구조를 기존 구조와 공존하게 만든다. (새 컬럼 추가)
2. **Migrate (이관)**: 기존 데이터를 새 구조로 복사한다. 이때 트리거나 앱 로직을 통해 실시간으로 생성되는 데이터가 양쪽 모두에 써지도록(`Dual Write`) 한다.
3. **Validate (검증)**: 새 구조의 데이터가 정확한지 검증한다.
4. **Contract (수축)**: 모든 데이터 이관이 확인되면 기존 로직과 컬럼을 제거한다.

이 과정은 번거롭지만, 사용자에게 단 1초의 중단도 허용하지 않아야 하는 대규모 서비스에서는 선택이 아닌 필수다.

### 7. 예제 코드로 보는 마이그레이션

최근 많이 사용하는 `Prisma` 나 `Drizzle` 같은 도구들은 이를 선언적으로 처리하지만, 본질을 이해하기 위해 원시적인 `SQL` 과 스크립트 형태의 흐름을 살펴보자.

```sql
-- 마이그레이션 파일 예시 (SQL)
-- 1. 새로운 컬럼 추가
ALTER TABLE users ADD COLUMN first_name VARCHAR(255);
ALTER TABLE users ADD COLUMN last_name VARCHAR(255);

-- 2. 데이터 이관
-- PostgreSQL의 경우 split_part 함수를 활용할 수 있다.
UPDATE users 
SET 
  first_name = split_part(full_name, ' ', 1),
  last_name = split_part(full_name, ' ', 2)
WHERE full_name IS NOT NULL;

-- 3. 제약 조건 추가 (데이터 이관 완료 후)
ALTER TABLE users ALTER COLUMN first_name SET NOT NULL;
```

만약 `Node.js` 환경에서 복잡한 정규식이나 외부 API 호출이 필요한 로직이 포함된다면 아래와 같은 스크립트를 작성하기도 한다.

```javascript
// migration_script.js (Node.js + Prisma 예시)
async function migrateUsers() {
  const users = await prisma.user.findMany({
    where: { first_name: null } // 아직 이관되지 않은 데이터만 조회
  });

  for (const user of users) {
    const parts = user.full_name.split(' ');
    const first = parts[0];
    const last = parts.slice(1).join(' ');
    
    await prisma.user.update({
      where: { id: user.id },
      data: { 
        first_name: first,
        last_name: last || ''
      }
    });
    console.log(`Migrated user: ${user.id}`);
  }
}
```

### 8. 성공적인 마이그레이션을 위한 팁 (Best Practices)

`마이그레이션` 에서 가장 중요한 건 '기술'보다 '절차'다.

1. **무조건 백업**: `마이그레이션` 직전의 `DB` 스냅샷은 생명줄이다. 복구 시나리오가 없는 `마이그레이션` 은 자살 행위나 다름없다.
2. **멱등성(Idempotency) 확보**: `마이그레이션` 스크립트를 실수로 두 번 실행하더라도 결과가 같아야 한다. "이미 존재하면 무시" 하거나 "이미 처리된 데이터는 스킵" 하는 로직이 필수다.
3. **스테이징 테스트**: 실제 운영 `DB` 와 동일한 규모와 구조의 데이터를 가진 스테이징 환경에서 반드시 먼저 실행해 봐야 한다. 운영에서만 발생하는 타임아웃이나 예외 상황을 미리 잡아야 한다.
4. **작게 쪼개기**: 한 번의 `마이그레이션` 에 너무 많은 변경을 담지 마라. 문제가 생겼을 때 원인을 찾기 힘들고 롤백도 어려워진다. "작은 성공의 반복"이 `마이그레이션` 의 진리다.

### 9. 결론: 마이그레이션은 선택이 아닌 필수다

초기 스타트업이나 개인 프로젝트에서는 "그냥 DB 밀고 새로 만들지 뭐"라는 생각이 들 수도 있다. 하지만 서비스가 커지고 단 한 명의 사용자라도 소중해지는 순간, `마이그레이션` 실력은 곧 서비스의 생존 능력과 직결된다.

기획은 언제나 바뀐다. 구현 방법도 날마다 진보한다. 어제의 코드를 부끄러워할 필요는 없지만, 어제의 데이터를 방치해서는 안 된다. 변화를 두려워하지 않고 유연하게 대응할 수 있는 시스템을 만드는 것, 그 핵심에 `마이그레이션` 이 있다.

데이터는 과거의 기록이지만, 그것을 어떻게 현재로 옮겨오느냐에 따라 서비스의 미래가 결정된다. 오늘 당신의 `DB` 에 새로운 컬럼을 추가할 때, 그것이 단순한 코드 한 줄이 아니라 사용자의 소중한 추억을 새 집으로 이사시키는 과정임을 기억하자.

> "코드의 변화는 깃(Git)이 책임지지만, 데이터의 변화는 마이그레이션이 책임진다. 그리고 그 모든 변화의 책임은 개발자에게 있다."
