## 1. '최종본'이라는 환상을 실체로 만드는 방법

디자인과 개발 사이에는 늘 보이지 않는 벽이 존재한다. 디자이너가 작업하는 캔버스는 끊임없이 변하는 생물과 같고, 개발자가 구현해야 하는 코드는 명확한 박제를 원하기 때문이다. `Figma`가 대세가 된 지금도 이 간극은 여전히 숙제로 남아있다. 피그마의 실시간 협업은 강력하지만, 개발자가 코드를 짜는 도중에 디자인이 슬쩍 바뀌어버리거나 "어느 레이어가 진짜 최종인가요?"라고 물어야 하는 상황이 빈번하기 때문이다.

이 지점에서 `Zeplin`은 독보적인 위치를 차지했다. 제플린은 디자인 툴이 아니라, 철저하게 디자인을 개발 언어로 번역하고 고정하는 데 특화된 공간이다. 디자이너가 스케치(Sketch)나 피그마에서 작업한 결과물을 제플린에 **퍼블리시(Publish)** 하는 순간, 그 화면은 개발자를 위한 공식적인 **명세(Spec)** 가 된다. 실시간으로 변하는 디자인 캔버스에서 벗어나, 개발자가 안심하고 집중할 수 있는 안정적인 환경을 제공하는 것. 이것이 제플린이 가진 첫 번째 가치였다.

## 2. 개발자가 원하는 모든 정보가 있는 '스펙 창고'

열악했던 시절에는 디자인 수치를 알기 위해 스크린샷 위에 자를 대거나 PDF를 뒤져야 했다. 하지만 제플린의 등장 이후, 개발자의 워크플로우는 완전히 바뀌었다. 제플린에 업로드된 시안의 요소들을 클릭하기만 하면, 각 기기별 해상도에 최적화된 마진과 패딩, 폰트 크기, 컬러 코드가 쏟아져 나왔다.

특히 이미지 에셋 관리는 그야말로 혁명이었다. 개발자가 디자이너에게 일일이 `@2x`, `@3x` 이미지를 요청할 필요가 없어졌다. 디자이너가 'Export' 설정만 해두면, 제플린은 개발자가 원하는 확장자(PNG, SVG, WebP, PDF 등)와 해상도별 레티나 사이즈를 알아서 생성해준다. 클릭 한 번으로 모든 배율의 이미지를 한꺼번에 다운로드할 수 있는 이 기능은, 수천 개의 아이콘을 다뤄야 하는 대규모 앱 개발 프로젝트에서 생산성을 수십 배 이상 끌어올렸다.

## 3. 시뮬레이터 오버레이: 1픽셀의 오차도 허용하지 않는 완벽주의

제플린의 기능 중 가장 '매니악'하면서도 강력한 기능은 단연 '시뮬레이터 오버레이(Simulator Overlay)'다. 이는 내가 개발한 앱의 실제 화면 위에 제플린의 디자인 시안을 투명한 레이어로 얹어서 비교해보는 기능이다.

개발자들은 종종 "눈으로 보기엔 똑같은데 왜 디자이너는 다르다고 할까?"라는 의문을 갖는다. 하지만 이 오버레이 기능을 켜고 투명도를 조절해보면, 1~2픽셀 어긋난 마진이나 미세하게 다른 폰트의 행간이 여과 없이 드러난다. 이 기능은 단순히 오차를 찾는 도구를 넘어, 개발자가 자신의 결과물에 대해 확신을 갖게 해주는 기준점이 되었다. 디자인 시안과 내 코드가 완벽하게 겹쳐졌을 때의 그 짜릿함은, 픽셀 단위의 완벽함을 추구하는 개발자(it's me)에게 최고의 보상이었다.

## 4. Figma vs Zeplin: 통합과 전문화의 차이

피그마가 기획부터 디자인까지 다 되는 올인원 툴이라면, 제플린은 디자인 핸드오프라는 외길을 걷는 전문 툴이다.

### Figma의 장단점
- **장점**: 모든 과정이 하나의 캔버스에서 일어난다. 소통이 빠르고 별도의 업로드 과정이 필요 없다.
- **단점**: 개발자가 디자인을 실수로 수정할 위험이 있고, 최종본이 파편화되기 쉽다. 명세 확인 인터페이스가 다소 복잡하게 느껴질 수 있다.

### Zeplin의 장단점
- **장점**: 디자인과 개발 환경이 명확히 분리되어 있어 안정적이다. 프로젝트와 스타일 가이드 개념이 견고하여, 디자인 시스템을 코드로 이식하기에 더 직관적이다. 개발 친화적인 코드 스니펫 기능이 매우 강력하다.
- **단점**: 디자인이 바뀔 때마다 다시 업로드해야 하는 번거로움이 있다. 비용적인 측면에서도 툴을 하나 더 써야 한다는 부담이 존재한다.

## 5. 디자인을 코드로 (Swift 예시)

제플린은 선택한 요소의 스타일을 곧바로 코드로 보여준다. 단순히 수치를 나열하는 게 아니라, 프로젝트 설정에 맞춘 Swift 코드를 생성해준다.

```swift
// Zeplin에서 제공하는 스타일 가이드를 활용한 Swift 코드 예시
import UIKit

class ZeplinStyleButton: UIButton {
    override init(frame: CGRect) {
        super.init(frame: frame)
        setupStyle()
    }
    
    required init?(coder: NSCoder) {
        super.init(coder: coder)
        setupStyle()
    }
    
    private func setupStyle() {
        // 제플린의 스타일 정보를 바탕으로 구현된 수치들
        self.backgroundColor = UIColor(red: 0.12, green: 0.45, blue: 0.98, alpha: 1.0)
        self.layer.cornerRadius = 8.0
        
        // Typography
        self.setTitleColor(.white, for: .normal)
        self.titleLabel?.font = UIFont.systemFont(ofSize: 16, weight: .bold)
        
        // Shadow (Zeplin 스펙 참조)
        self.layer.shadowColor = UIColor(red: 0, green: 0, blue: 0, alpha: 0.15).cgColor
        self.layer.shadowOffset = CGSize(width: 0, height: 4)
        self.layer.shadowRadius = 12
        self.layer.shadowOpacity = 1
    }
}
```

이런 코드 스니펫은 개발자가 속성 하나하나를 타이핑하는 수고를 덜어줄 뿐만 아니라, 디자인 토큰(Design Tokens)을 코드와 일치시키는 데 큰 도움을 준다.

## 6. 마치며: 소통의 본질은 도구 너머에 있다

우리는 매일 새로운 툴을 만나고 이별한다. 제플린의 전성기가 예전만 못하다는 이야기도 들리지만, 그들이 추구했던 '디자이너와 개발자 사이의 명확하고 친절한 대화'라는 가치는 피그마를 비롯한 모든 현대적 협업 툴의 뿌리가 되었다.

결국 중요한 것은 어떤 툴을 사용하느냐가 아니라, 서로의 작업 영역을 존중하고 더 나은 결과물을 위해 끊임없이 소통하려는 마음가짐이다. 제플린은 그 마음가짐을 가장 구체적이고 전문적인 인터페이스로 보여준 툴이었다. 우리가 1픽셀의 오차에 집착하며 시뮬레이터를 겹쳐보던 그 치열했던 시간들이 모여, 지금의 더 나은 앱 서비스들이 만들어지고 있는 것이 아닐까.

> "디자인은 개발을 통해 생명력을 얻고, 개발은 디자인을 통해 완성도를 갖춘다. 제플린은 그 둘 사이를 잇는 가장 단단한 매듭이었다."
