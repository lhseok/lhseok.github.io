개발자로 살다 보면 가장 귀찮은 순간 중 하나가 바로 테스트 코드를 짤 때다. "기능 구현하기도 바빠 죽겠는데, 테스트까지 짜야 하나?"라는 생각이 머릿속을 떠나지 않는다. 하지만 역설적으로, 그 귀찮음을 이겨내고 짠 **테스트 케이스(Test Case)** 가 내 커리어를 구했던 순간들이 있었다. 오늘은 테스트 케이스가 왜 단순한 서류 작업이 아닌, 개발자의 생존 전략인지 이야기해보려 한다.

![Software testing concept](https://images.unsplash.com/photo-1516259762381-22954d7d3ad2?q=80&w=1280&auto=format&fit=crop!!ar=3:1)

### 1. 테스트 케이스란 무엇인가?

쉽게 말해 테스트 케이스는 **특정 기능이 의도대로 동작하는지 확인하기 위한 시나리오** 다. 입력값, 실행 조건, 그리고 우리가 기대하는 결과값을 세트로 묶은 것이다. 단순히 "로그인이 잘 된다"가 아니라, "아이디에 공백이 포함되었을 때 에러 메시지가 출력되는가?"처럼 아주 구체적이고 쪼개진 단위의 확인 절차를 의미한다.

실무에서 테스트 케이스는 크게 세 가지 구성 요소를 갖춘다.
1. **Pre-condition (사전 조건)**: 테스트를 수행하기 위해 시스템이 어떤 상태여야 하는가? (예: 사용자가 로그아웃된 상태)
2. **Steps (테스트 단계)**: 어떤 동작을 순서대로 수행하는가? (예: 아이디 입력 -> 비밀번호 입력 -> 로그인 버튼 클릭)
3. **Expected Result (기대 결과)**: 최종적으로 어떤 일이 일어나야 하는가? (예: 메인 화면으로 이동)

### 2. 왜 작성해야 하는가? (얻는 것 vs 잃는 것)

테스트 케이스를 작성하면 당장은 개발 속도가 느려지는 것처럼 보인다. 하지만 장기적으로는 무조건 이득이다. 소프트웨어는 살아있는 생명체와 같아서 끊임없이 변한다. 어제의 코드가 오늘 터지지 않으리라는 보장이 없다.

- **코드 리뷰와 리팩토링의 용기**: 코드를 대대적으로 수정했을 때, 기존 기능이 망가졌는지 일일이 손으로 눌러볼 순 없다. 이때 테스트 케이스는 "너의 수정은 안전해"라고 말해주는 유일한 친구다. 테스트 코드가 촘촘하게 짜여 있다면, 우리는 훨씬 더 과감하게 기술 부채를 해결하고 리팩토링을 감행할 수 있다.
- **문서로서의 역할**: 잘 짜인 테스트 케이스는 그 자체로 훌륭한 명세서가 된다. 이 함수가 어떤 예외 상황을 고려했는지 코드를 다 뜯어보지 않아도 알 수 있다. 새로 합류한 팀원이 코드를 이해하는 가장 빠른 방법은 테스트 코드를 읽어보는 것이다.
- **사이드 이펙트 방지**: A를 고쳤는데 뜬금없이 저 멀리 있는 B가 터지는 공포에서 벗어날 수 있다. 복잡한 시스템일수록 의존성이 얽혀 있어, 전혀 상관없어 보이는 곳에서 에러가 발생하기 마련이다. 자동화된 테스트 케이스는 이런 '사이드 이펙트'를 즉각적으로 잡아낸다.

반면 테스트 케이스가 없다면? 실무에서는 이런 일이 비일비재하다. 신규 기능을 배포했는데 갑자기 결제 버튼이 안 눌린다거나, 특정 OS 버전에서만 앱이 튕기는 식이다. 사람이 직접 확인하는 '수동 테스트'는 한계가 명확하고, 피로도에 따라 구멍이 숭숭 뚫리기 마련이다. 특히 마이너한 엣지 케이스(Edge Case)들은 수동 테스트에서 발견될 확률이 거의 제로에 가깝다.

### 3. 실무 사례: 이벤트 1시간 전, 나를 구한 테스트 케이스

실제 경험담이다. 예전 프로젝트에서 연중 가장 큰 규모의 할인 이벤트를 앞두고 있었다. 이벤트 시작 직전, 마케팅 팀의 요청으로 쿠폰 적용 로직에 아주 미세한 수정을 가했다. 겉보기엔 아무 문제 없었고, 기획자도 수동 테스트 후 "OK"를 외쳤다. 눈으로 보기엔 할인 금액이 잘 계산되는 것처럼 보였기 때문이다.

하지만 혹시나 하는 마음에 전체 테스트 스위트를 돌려보았다. 그리고 잠시 후, **유닛 테스트(Unit Test)** 에서 빨간불이 들어왔다. 특정 등급의 사용자가 쿠폰을 중복으로 적용할 수 있는 로직 에러가 발견된 것이다. 이 에러는 특정 결제 수단과 특정 쿠폰 타입이 만났을 때만 발생하는 아주 희귀한 조합이었다.

만약 이대로 배포되었다면 수억 원대의 손실이 발생할 뻔했다. 이벤트 시작 30분 전, 우리는 식은땀을 흘리며 코드를 수정했고 다시 테스트가 통과하는 것을 확인한 뒤 무사히 배포를 마쳤다. 테스트 케이스가 없었다면 나는 아마 사유서를 쓰고 있었거나, 서비스 장애로 밤을 지새우고 있었을 거다.

### 4. Swift 기반 프로젝트에서의 테스트 케이스 작성 요령

iOS 개발 환경(Swift)에서는 주로 `XCTest` 프레임워크를 사용한다. 효과적인 테스트를 위해선 몇 가지 실전 요령이 필요하다.

1. **Given-When-Then 패턴**: 테스트의 구조를 명확히 한다.
   - **Given**: 테스트 환경 설정 (객체 생성, Mock 데이터 준비)
   - **When**: 실제 동작 수행 (함수 호출)
   - **Then**: 결과 검증 (Assertion)
2. **독립성 유지 (Isolate)**: 테스트 하나가 다른 테스트의 결과에 영향을 주면 안 된다. 각 테스트 메서드는 독립적으로 실행될 수 있어야 하며, DB나 네트워크 같은 외부 의존성은 Mock 객체로 대체하는 것이 기본이다.
3. **네이밍은 구체적으로**: 단순히 `testLogin()` 보다는 `testLogin_WithEmptyPassword_ShouldShowErrorMessage()` 처럼 어떤 조건에서 어떤 결과가 나와야 하는지 함수 이름만 보고도 알 수 있게 짠다.
4. **빠른 피드백**: 테스트는 빨라야 한다. 전체 테스트를 돌리는 데 10분 이상 걸린다면 개발자는 테스트를 돌리는 걸 꺼리게 된다. 불필요한 네트워크 통신이나 과도한 데이터 생성을 피해야 한다.

### 5. 예시 코드로 보는 테스트 프로세스

간단한 포인트 계산 로직을 테스트하는 예시 코드다. 실제 커머스 앱에서 흔히 볼 수 있는 로직을 가정했다.

```swift
import XCTest
@testable import MyApp

final class PointCalculatorTests: XCTestCase {
    var sut: PointCalculator! // System Under Test

    override func setUp() {
        super.setUp()
        // 각 테스트가 시작되기 전에 항상 신선한 상태로 초기화
        sut = PointCalculator()
    }

    override func tearDown() {
        sut = nil
        super.tearDown()
    }

    func testCalculatePoints_WhenAmountIs1000_ShouldReturn10() {
        // Given (준비): 결제 금액이 1000원인 상황
        let amount = 1000
        
        // When (실행): 포인트 계산 함수 호출
        let result = sut.calculate(from: amount)
        
        // Then (검증): 결과가 10점인지 확인
        XCTAssertEqual(result, 10, "1000원 결제 시 포인트는 1%인 10점이어야 함")
    }
    
    func testCalculatePoints_WhenAmountIsNegative_ShouldReturnZero() {
        // Given: 결제 금액이 음수인 비정상적인 상황
        let amount = -500
        
        // When
        let result = sut.calculate(from: amount)
        
        // Then: 음수 금액에 대해서는 0점을 반환해야 함
        XCTAssertEqual(result, 0, "음수 금액에 대해서는 포인트를 부여하지 않아야 함")
    }
}
```

이런 식으로 작성된 코드는 CI/CD 파이프라인(예: GitHub Actions)에 묶어두면 좋다. 코드가 Merge 되기 전에 자동으로 이 테스트들이 실행되도록 설정하면, 휴먼 에러가 실제 서비스에 반영되는 것을 원천 차단할 수 있다.

### 6. 결론: 미래의 나를 위한 가장 저렴한 보험

테스트 케이스를 작성하는 시간은 낭비가 아니라 보험료를 내는 것과 같다. 당장은 아깝지만, 사고가 났을 때 나를 지켜줄 유일한 수단이기 때문이다. 숙련된 개발자일수록 자신의 직관보다 시스템의 검증을 믿는다.

내가 짠 코드를 100% 믿지 마라. 대신 내가 짠 테스트 케이스를 믿어라. 오늘 내가 귀찮아서 생략한 테스트 케이스 하나가, 내일의 내 주말을 앗아갈 수도 있다는 사실을 명심하자.

> "테스트 케이스는 가장 신뢰할 수 있는 동료이자, 제품의 생명력을 연장하는 최후의 방어선이다."
