클로드를 쓰다 보면 같은 말을 반복하게 된다. "우리 팀 코드 컨벤션은 이렇다", "블로그 글은 이런 구조로 써라" 같은 것들이다. **`프로젝트(Projects)`** 에 지침을 넣을 수도 있지만, 모든 채팅에서 공통으로 필요한 '능력'이라면 **`Skills`** 를 쓰는 게 훨씬 효율적이다.

![Tools on a wooden table](https://images.unsplash.com/photo-1581092160607-ee22621dd758?q=80&w=1280&auto=format&fit=crop!!ar=2.5:1)

### 1. Claude Skills가 뭐야?

쉽게 말해 클로드에게 **'기술'** 을 가르치는 기능이다. 단순한 텍스트 프롬프트와 다른 점은 클로드가 상황을 보고 "지금 이 스킬이 필요하겠네" 라고 판단해서 능동적으로 가져다 쓴다는 점이다. 2025년 말에 정식으로 도입된 이 기능은 단순 지식 공유를 넘어선 **'워크플로우 자동화'** 에 가깝다.

- **프롬프트**: 대화할 때마다 일일이 설명해야 함. 컨텍스트가 길어지면 잊어버리기도 함.
- **Skills**: 한 번 정의해두면 클로드가 알아서 호출함. 필요한 시점에만 로드되어 효율적임.

**`MCP (Model Context Protocol)`** 가 클로드에게 외부 데이터에 접근할 수 있는 '통로'를 열어주는 거라면, **`Skills`** 는 그 통로를 어떻게 이용할지, 혹은 특정 작업을 어떤 순서로 처리할지 알려주는 **'레시피'** 역할을 한다.

### 2. 왜 쓰는가? (장단점)

**장점**
- **컨텍스트 최적화**: 스킬의 전체 내용을 항상 컨텍스트에 들고 있지 않는다. 필요할 때만 로드하므로 불필요한 토큰 낭비를 줄이고 '컨텍스트 윈도우'를 깨끗하게 유지한다.
- **일관된 결과**: 복잡한 작업(예: 데이터 분석 후 보고서 작성)을 수행할 때 단계를 빼먹지 않고 항상 같은 퀄리티를 보장한다.
- **실행 능력**: 스킬 내부에 실행 가능한 코드(Python 등)나 특정 템플릿을 포함할 수 있어 단순 텍스트 생성을 넘어선 작업이 가능하다.

**단점**
- **초기 설계 비용**: 스킬이 제대로 작동하려면 예외 상황까지 고려한 촘촘한 설계가 필요하다.
- **업데이트 관리**: 업무 프로세스가 바뀌면 스킬 파일도 직접 수정해줘야 한다. 안 그러면 낡은 지식을 기반으로 작업한다.

### 3. 작동 원리: 단계적 노출 (Progressive Disclosure)

클로드는 효율성을 위해 스킬을 3단계로 관리한다.

1. **상시 대기 (System Prompt)**: 스킬의 이름과 "이 스킬은 언제 쓰인다" 는 설명만 로드된다.
2. **조건부 로드 (SKILL.md)**: 사용자의 요청이 스킬 설명과 일치하면, 그제서야 **`SKILL.md`** 에 적힌 상세 워크플로우를 읽어들인다.
3. **실행 시점 로드 (Linked Assets)**: 스킬에 포함된 파이썬 스크립트나 대용량 데이터는 실제 실행 버튼을 누르는 순간에만 컨텍스트에 포함된다.

### 4. 실무 예시: 팀 전용 코드 리뷰 스킬

단순한 오타 검수를 넘어, 실제 팀의 컨벤션을 강제하고 보안 취약점까지 체크하는 스킬을 만든다고 가정해보자.

**`SKILL.md` 정의 (예시)**
```markdown
# 테크 팀 코드 리뷰 스킬

## 목표
GitHub PR 생성 전, 팀의 보안 표준과 코드 스타일을 검수하여 일관성을 유지한다.

## 검수 항목 (우선순위 순)
1. **보안**: API Key, DB 비밀번호 등 민감 정보가 하드코딩 되어 있는가?
2. **비즈니스 로직**: 결제 관련 로직에서 트랜잭션 처리가 누락되지 않았는가?
3. **스타일**: 변수명은 camelCase를 준수하며, 함수 하나가 50줄을 넘지 않는가?
4. **테스트**: 신규 기능에 대한 단위 테스트 코드가 포함되어 있는가?

## 워크플로우
1. 변경된 코드 파일을 분석하여 위 4가지 항목에 대해 점수를 매긴다. (10점 만점)
2. 7점 미만인 항목이 있다면 즉시 '반려(Reject)' 사유와 수정 가이드를 제시한다.
3. 모든 항목이 통과되면 PR에 바로 올릴 수 있는 '요약 코멘트'를 작성한다.

## 제약 사항
- 의견은 항상 비판적이되 건설적이어야 한다.
- "좋네요" 같은 의미 없는 칭찬은 생략하고 개선점에 집중한다.
- 코드 예시는 반드시 팀의 표준 라이브러리를 사용한다.
```

이렇게 정의된 파일을 클로드 설정의 **`Skills`** 에 업로드하거나, **`Claude Code`** 라면 **`.claude/skills/`** 폴더에 넣는다. 이제 "이 파일 리뷰해줘" 라고 하면 클로드는 시스템에 주입된 이 스킬을 꺼내어 팀 시니어 개발자처럼 깐깐하게 코드를 털어준다.

### 5. 직접 만드는 법 (Skill Creator 활용)

처음부터 마크다운을 짜기 막막하다면 클로드에게 직접 시키면 된다.

1. 클로드에게 "특정 작업을 수행하는 스킬을 만들고 싶어" 라고 말한다.
2. 클로드 내장 도구인 **`skill-creator`** 가 작동하며 필요한 질문을 던진다.
3. 질문에 답하면 클로드가 **`SKILL.md`** 와 필요한 스크립트 구조를 짜준다.
4. 완성된 폴더를 압축해서 클로드 설정에서 활성화한다.

### 6. 유의사항 및 팁

스킬은 만능 도구가 아니다. 효과를 극대화하려면 다음을 기억해야 한다.

- **슬래시 명령어 활용**: 클로드가 가끔 스킬을 안 꺼낼 때가 있다. 그럴 땐 `/review` 처럼 슬래시 명령어로 강제 호출하면 확실하다.
- **모듈화**: 한 스킬에 너무 많은 기능을 넣지 마라. '코드 리뷰', '문서 작성', '데이터 변환' 등으로 잘게 쪼개는 게 호출 정확도가 높다.
- **피드백 루프**: 스킬을 써보고 결과가 맘에 안 들면 "이 부분은 스킬 지침을 좀 더 강화해줘" 라고 말해서 **`SKILL.md`** 를 계속 업데이트해야 한다.

결국 스킬을 잘 쓴다는 건 클로드에게 일을 시키는 단계를 표준화한다는 뜻이다. 단순히 말을 잘 알아듣는 AI를 넘어, 내 업무 방식이 이식된 **'디지털 복제본'** 을 만드는 과정이다. 반복적인 지시에 지쳤다면 당장 스킬부터 정의해보길 권한다.

> "AI가 똑똑해지는 건 모델 탓이 아니라, 우리가 건네주는 작업 지침의 밀도 덕분이다. 스킬은 그 밀도를 만드는 가장 영리한 방법이다."
