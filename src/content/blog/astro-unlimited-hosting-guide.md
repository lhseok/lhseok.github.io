Astro로 블로그를 만들고 글을 하나둘 쌓다 보면 즐거움도 잠시, 현실적인 벽에 부딪히는 순간이 온다. 바로 **용량 제한**과 **파일 개수 제한**이다. "내 글 몇 개 안 되는데 벌써?" 싶겠지만, 고화질 이미지 몇 장 섞인 포스팅이 수십 개만 쌓여도 무료 호스팅 서비스들은 슬슬 눈치를 주기 시작한다.

나 역시 GitHub Pages부터 시작해 Cloudflare Pages, Vercel, Netlify를 모두 떠돌아다녔다. 이 여정 끝에 내가 정착한, 소위 '끝판왕' 아키텍처를 소개하려고 한다. 바로 **[GitHub Private + Cloudflare Pages + R2]** 조합이다. 왜 이 조합이어야 했는지, 그리고 어떻게 구축하는지 상세히 기록해 본다.

---

### 1. 여정의 기록: 왜 그들은 내 블로그의 종착지가 되지 못했나?

블로그 아키텍처를 결정하기까지 검토했던 플랫폼들의 장단점과 뼈아픈 한계점들을 정리해 봤다.

#### ❶ GitHub Pages: "내 소스코드를 다 보여줘야 한다고?"
가장 접근성이 좋지만, 치명적인 약점이 있다.
*   **프라이빗의 유료화**: 무료 계정에서 GitHub Pages를 쓰려면 레포지토리를 **Public**으로 설정해야 한다. 블로그 글의 초안이나 설정 파일 등이 전 세계에 생중계되는 셈이다.
*   **용량 제한**: 전체 사이트 용량이 **1GB**를 넘으면 배포가 차단된다. 이미지가 많은 블로그라면 생각보다 금방 도달하는 수치다.

#### ❷ Cloudflare Pages (초기): "2만 개라는 보이지 않는 벽"
GitHub 없이 CLI 명령(`wrangler`)으로 바로 배포할 수 있고 대역폭도 무제한이라 최고인 줄 알았다. 하지만 문서를 꼼꼼히 읽다 발견한 사실 하나.
*   **파일 개수 제한**: 프로젝트당 **총 파일 개수가 2만 개**로 제한되어 있다. Astro는 빌드 시 정적 파일을 생성하는데, 글이 늘어나고 이미지가 쌓이다 보면 수년 내에 이 제한에 걸려 블로그가 '동결'될 위험이 있었다.

#### ❸ Vercel vs Netlify: "트래픽 초과 과금의 압박"
파일 개수 제한이 넉넉한 타 플랫폼들도 당연히 검토 대상이었다.

| 비교 항목 | Vercel (버셀) | Netlify (넷리파이) |
| :--- | :--- | :--- |
| **파일 개수** | 하드웨어적 제한 사실상 없음 | 사실상 없음 |
| **무료 트래픽** | 월 100GB 제공 | 월 100GB 제공 |
| **주의 사항** | **비상업적 용도만 무료**. (애드센스 달면 유료 대상) | 빌드 속도가 다소 느리고 트래픽 초과 시 비용 발생 |

결국 이들은 '**트래픽 제한**'이라는 또 다른 벽을 가지고 있었다. 방문자가 늘어나는 건 기쁜 일인데, 그게 비용으로 돌아온다면 블로그 운영이 부담스러워질 수밖에 없다.

---

### 2. 최종 결론: 왜 [Cloudflare Pages + R2]가 정답인가?

모든 플랫폼의 장점만 취하고 단점은 무력화하는 최종 아키텍처의 핵심은 '**분리**'에 있다.

1.  **GitHub Private 레포지토리**: 소스 코드는 나만 볼 수 있게 숨기고, 안전하게 백업한다.
2.  **이미지는 Cloudflare R2로 격리**: 파일 개수와 용량을 가장 많이 잡아먹는 이미지를 Astro 프로젝트 내부(`public/images` 등)에서 완전히 제외한다. R2는 **송신 트래픽(Egress) 요금이 무료**라 방문자가 폭발해도 돈이 한 푼 안 든다.
3.  **Cloudflare Pages 호스팅**: 이미지를 R2로 뺐으니 Astro가 빌드하는 건 순수 HTML과 가벼운 JS뿐이다. '**파일 2만 개 제한**'은 이제 평생 걸릴 일 없는 숫자가 된다. Pages의 압도적인 속도와 무제한 트래픽 혜택만 챙기면 끝이다.

### 3. 실전 구축 가이드 (Step-by-Step)

#### Step 1: GitHub Private 레포지토리 세팅
먼저 GitHub에서 **Private**으로 새 레포지토리를 만들자.

```bash
git init
git add .
git commit -m "feat: 무제한 블로그 시스템 초안"
git branch -M main
git remote add origin https://github.com/[사용자명]/[레포이름].git
git push -u origin main
```

#### Step 2: Cloudflare Pages 자동 배포 연동
이제 깃허브에 글을 올리기만 하면 클라우드플레어가 알아서 빌드하도록 연결한다.
1.  Cloudflare 대시보드 → **Workers & Pages** → **Create** → **Pages** 탭 클릭.
2.  **Connect to Git**을 선택하고 방금 만든 Private 레포지토리를 연결한다.
3.  Framework preset에서 **Astro**를 선택하면 빌드 명령어와 디렉토리가 자동으로 세팅된다.

#### Step 3: Cloudflare R2 이미지 저장소 세팅
1.  Cloudflare 대시보드 → **R2** → **Create bucket** 클릭.
2.  버킷 이름을 정하고(예: `blog-assets`) 생성한다.
3.  생성된 버킷의 **Settings** 탭 → **Public Access** 항목에서 **R2.dev Subdomain**을 활성화한다. 이제 이미지가 웹상에 공개될 수 있는 고유 주소(`https://pub-xxxx.r2.dev`)가 발급된다.

---

### 4. 무제한 블로그를 즐기는 실전 워크플로우

세팅이 끝났다면 평소 블로그 운영은 이렇게 하면 된다.

1.  **이미지 업로드**: 포스팅에 쓸 사진을 R2 대시보드에 드래그하여 올린다. 그리고 해당 파일의 공개 URL을 복사한다.
2.  **마크다운 작성**: 프로젝트 내부 `public` 폴더에 이미지를 절대 넣지 마라. 대신 본문에 R2 링크를 넣는다.

```markdown
---
title: "용량 제한을 초월한 첫 번째 포스팅"
pubDate: "2026-05-25"
---

아래 이미지는 프로젝트 파일 개수에 영향을 주지 않는 Cloudflare R2에서 로딩됩니다.

![여행 사진](https://pub-xxxx.r2.dev/2026/my-travel-photo.png)
```

3.  **한 줄 배포**: 글 작성이 끝났다면 터미널에 이 명령 한 줄만 입력하고 노트북을 닫으면 된다.

```bash
git add . && git commit -m "post: 새 글 추가" && git push
```

`push`가 되는 순간 내 원본 글은 Private 레포에 백업되고, Cloudflare Pages는 이를 감지해 약 1분 뒤 전 세계에 새 글을 라이브로 뿌려준다.
