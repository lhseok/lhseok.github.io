예전에는 디자인이 나오면 무조건 PDF로 받았다. `PDF` 파일 하나에 화면 전체가 있고, 그 옆에 조그맣게 폰트 사이즈, 간격, 컬러 코드가 적혀 있었다. 디자이너가 수기로 하다보니 당연히 기재가 누락되는 경우가 다반사였고, 개발자는 PDF를 띄워놓고 픽셀을 일일이 재기도 했고 심지어 눈으로 가늠해서 상상 코딩하는 경우도 있었다. 앱 배포 이후 디자이너에게 늘 듣는 말: "여기 간격이 다른거 같은데 확인 부탁드려요." 거기에 개발자는 "PDF 그대로 했는데요."라고 할 수 밖에 없었다. 그 시절 디자인 명세와 실제 코드 사이의 괴리는 개발자가 짊어져야 할 피로의 핵심이었다.

![PDF 이전 인쇄물에 펜으로 기재했던 시절이 있었다면 믿겠는가.](https://images.unsplash.com/photo-1576153192396-180ecef2a715?q=80&w=1280&auto=format&fit=crop!!ar=2.5:1)

### 1. Figma가 바꾼 것들

그러다 `Figma` 가 등장했다. `Figma` 는 웹 기반이라 디자이너와 개발자가 같은 문서를 동시에 보고 수정할 수 있다는 점에서 충격적이었다. 더 이상 PDF를 주고받을 필요가 없었다. 개발자가 직접 `Figma` 에 들어가 레이어를 클릭하고 디자인 수치를 확인할 수 있게 됐다.

`Figma` 로 넘어오면서 디자인-개발 간의 커뮤니케이션은 확실히 빨라졌다. 디자이너가 수정한 내용이 실시간으로 보이고, 개발자는 명확한 수치를 복사할 수 있으니 픽셀 단위로 싸울 일이 줄었다. 버전 관리도 훨씬 수월해졌다. 과거에는 PDF 파일 이름이 `homepage_final_v2_really_final.pdf`가 되기 일쑤였는데, `Figma` 는 히스토리 기능을 통해 변경 사항을 추적할 수 있었다.

하지만 여전히 해결되지 않은 문제가 있었다. 바로 코딩 그 자체였다. 디자인 레이어를 확인하고, 레이아웃을 구성하고, 스타일을 입히고, 상태에 따른 비즈니스 로직을 짜는 일은 여전히 개발자의 순수한 노동력이 필요했다. 디자인이 복잡해질수록 이 과정은 지루하고 반복적인 작업으로 변해갔다.

### 2. AI가 연결하는 디자인과 코드

지금은 상황이 다르다. `Claude` 와 같은 `LLM` 이 디자인의 의도를 읽어내기 시작했다. 이제 `Figma` 의 디자인 데이터를 `Claude Code` 에 던져주면, AI가 알아서 네이티브 UI 코드를 짜준다. 단순히 값을 복사하는 수준이 아니라, 컴포넌트 구조를 이해하고, 상태 변화를 반영하며, 다양한 화면 크기까지 고려한다.

이 과정에서 가장 중요한 건 AI에게 디자인을 설명하는 방식이다. `Figma` 데이터를 어떻게 정리해서 넘기느냐에 따라 AI가 작성하는 코드의 품질이 결정된다. 단순히 레이어 정보를 전달하는 것이 아니라, 디자인 시스템의 토큰(Design Tokens)을 함께 전달해야 AI가 일관된 코드를 생성할 수 있다.

### 3. 디자인을 코드로 바로 만들기: 실무 워크플로우

디자인을 AI로 즉시 코드로 구현하는 과정은 단순히 명령을 내리는 것을 넘어, 체계적인 명세 전달 과정이 필요하다.

#### 1단계: 디자인 시스템 기반의 Figma 구조화
`Figma` 파일이 엉망이면 AI도 코드를 엉망으로 만든다. Auto Layout을 제대로 쓰고, 컴포넌트 이름을 명확하게 정의해야 한다.

- 컴포넌트 이름: `Button/Primary`, `Button/Secondary` 등
- 변수 활용: 컬러, 간격, 폰트 등을 `Figma Variables` 로 관리
- 디자인 토큰: 디자인 요소들을 코드에서 사용하는 변수명과 일치시킨다. (예: `color-primary-500` = `brand-color`)

#### 2단계: 디자인 명세(Spec)의 데이터화
추출한 데이터를 바탕으로 간단한 `JSON` 파일을 만든다. 이 파일이 AI의 가이드라인이 된다.

```json
{
  "button": {
    "padding": { "horizontal": 16, "vertical": 8 },
    "cornerRadius": 8,
    "colors": { "primary": "#000000", "text": "#FFFFFF" }
  }
}
```

#### 3단계: Claude Code에 명세와 맥락 주입
`Claude Code`에 단순히 "이거 코드로 짜줘"라고 하면 안 된다. 아래와 같이 구체적인 컨텍스트와 함께 명세를 제공해야 한다.

```bash
# Claude Code에 명세와 제약조건 주입
/chat "이 디자인 명세(spec.json)를 참조하여 SwiftUI의 PrimaryButton 뷰를 구현해줘.
- 이 컴포넌트는 SwiftUI 기반이다.
- 디자인 시스템에 맞게 네이티브 뷰 스타일을 적용한다.
- 상태 처리는 @State를 활용하고, 클릭시 비동기 액션을 실행하는 구조로 만들어줘.
- 제약 조건: 기존 팀의 버튼 컴포넌트 구조를 반드시 준수할 것."
```

#### 4단계: 코드 생성 및 검증
위와 같이 요청하면 `Claude`는 디자인 수치와 비즈니스 로직 요구사항을 정확히 조합하여 즉시 사용할 수 있는 코드를 생성해준다.

```swift
// Claude가 생성한 SwiftUI 버튼 컴포넌트 예시
import SwiftUI

struct PrimaryButton: View {
    let title: String
    let action: () async -> Void
    @State private var isLoading = false

    var body: some View {
        Button(action: {
            Task {
                isLoading = true
                await action()
                isLoading = false
            }
        }) {
            Text(isLoading ? "로딩중..." : title)
                .padding(.horizontal, 16)
                .padding(.vertical, 8)
                .background(isLoading ? Color.gray : Color.black)
                .foregroundColor(.white)
                .cornerRadius(8)
        }
        .disabled(isLoading)
    }
}
```

이 코드는 디자인 시스템의 수치를 정확히 반영하고 있으며, SwiftUI의 `Task`를 활용해 비동기 상태를 깔끔하게 처리하고 있다. 핵심은 `데이터화된 명세`와 `명확한 기술적 제약조건`을 동시에 제공하는 것이다.

### 4. 한계와 방향성

물론 AI가 만든 코드를 무조건 신뢰할 수는 없다. 특히 복잡한 애니메이션이나 정교한 상태 관리는 AI가 한 번에 완성하지 못할 때가 많다. 또한 AI가 디자인의 미묘한 의도(예: 디자이너가 의도한 여백의 긴장감)를 놓칠 때도 있다. 

이런 경우, 단순히 AI가 뱉어낸 코드를 그대로 쓰는 게 아니라, `Claude` 와 대화하며 수정해나가는 과정이 필수적이다. "여기 패딩을 좀 더 줄여줘", "이 상태에서 hover 효과가 어색해" 같은 피드백을 통해 코드를 점진적으로 개선해야 한다.

중요한 건 `Claude` 가 디자인-코드 간의 '번역'을 대신해줌으로써 개발자의 시간을 대폭 줄여줬다는 점이다.

이제 개발자는 픽셀을 재고, 디자인 수치를 확인하는 단순 노동에서 벗어나야 한다. 대신 `Figma` 를 더 구조적으로 설계하고, AI가 디자인 의도를 완벽히 이해할 수 있도록 명세의 밀도를 높이는 일에 집중해야 한다. 디자인과 코드 사이의 거리는 더 가까워질 것이고, 그 사이를 AI가 잇고 있다.

> "디자인은 이제 명세서가 아니라 코드로 가는 설계도이다. 그리고 AI는 그 설계도를 해석하는 가장 빠른 통역사가 되었다."
